## How It Works Now

### Settlement Process Flow

```
1. Manager initiates settlement
   ↓
2. Bill calculation
   - Subtotal
   - Tax (CGST/SGST)
   - Discounts
   - Charges
   - Loyalty redemption
   ↓
3. Invoice number generated (atomic)
   ↓
4. Order updated with:
   ✅ status = 'settled'
   ✅ settled_at = NOW()
   ✅ payment_status = 'paid'
   ✅ final_amount calculated
   ↓
5. Table freed
   - Sync table lifecycle
   - Set table.status = 'available'
   - Clear assignments
   ↓
6. Real-time events broadcast
   ✅ ORDER EVENT: order settled
   ✅ TABLE EVENT: table updated
   ↓
7. Connected clients receive SSE events
   - Order page updated immediately
   - Table card shows available
   - No page reload needed
   ✓ All staff see changes
```

## Troubleshooting

### Issue: Order status still shows 'completed'
**Solution:**
1. Restart backend server
2. Clear any caches
3. Verify orderService.js changes were applied
4. Check: `grep "status: 'settled'" backend/src/services/orderService.js`

### Issue: No real-time updates in UI
**Solution:**
1. Check SSE connection: Network tab in DevTools
2. Verify event broadcasts: Check backend logs for "table_updated"
3. Ensure restaurantId matches in broadcast
4. Check: `grep "broadcastRestaurantEvent" backend/src/services/orderService.js`

### Issue: Table not freeing
**Solution:**
1. Verify order.table_id is set
2. Check TableService.syncTableLifecycle is called
3. Inspect table status after settlement
4. Check backend logs for table sync errors

### Issue: Invoice number not generated
**Solution:**
1. Verify RPC function exists: `generate_bill_number`
2. Check database permissions
3. Review settlement logs for RPC errors
4. Test RPC directly if needed
